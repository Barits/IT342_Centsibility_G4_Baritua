# Phase 1 Implementation Summary
## User Registration and Login

**Date:** March 4, 2026  
**Project:** Centsibility Financial Management Platform  
**Phase:** 1 - User Registration and Login

## User Registration

**Registration Fields Used:**
- First Name (required, 2-50 characters)
- Last Name (required, 2-50 characters)
- Email (required, valid email format, unique)
- Password (required, minimum 8 characters)
- Confirm Password (must match password)

**Validation Process:**
The system performs two-level validation. Client-side validation uses Formik and Yup libraries to provide real-time feedback on form fields, checking for required fields, email format, and password complexity (uppercase, lowercase, digit, special character). Server-side validation uses Jakarta Bean Validation annotations (@NotBlank, @Email, @Size, @Pattern) to verify all inputs before processing, with a global exception handler to catch and format any validation errors.

**How Duplicate Accounts are Prevented:**
Duplicate accounts are prevented at two levels. The database has a UNIQUE constraint on the email column, preventing duplicate entries at the database level. The application also checks if an email exists using UserRepository.existsByEmail() before creating a new user. If a duplicate email is detected, the system returns a 409 Conflict status with an error message.

**How Passwords are Stored Securely:**
Passwords are hashed using BCrypt algorithm with 12 salt rounds through Spring Security's BCryptPasswordEncoder. Passwords are never stored in plain text. Each password gets a unique salt automatically generated by BCrypt, making rainbow table attacks ineffective.

## User Login

**Login Credentials Used:**
- Email (required, valid format)
- Password (required)

**How the System Verifies Users:**
When a user submits login credentials to /api/auth/login, Spring Security's AuthenticationManager handles the authentication process. The UserDetailsService loads the user from the database by email. The system then verifies the submitted password against the stored BCrypt hash using secure comparison. If credentials are valid, a JWT token is generated and returned.

**What Happens After Successful Login:**
Upon successful login, the user receives a JWT token and their user data in the response. The frontend stores the JWT token in localStorage as 'accessToken' and stores user data for display purposes. The user is automatically redirected to the dashboard. All subsequent API requests include the JWT token in the Authorization header, which the backend validates before processing protected endpoints.

## Database Table

**Table:** users

**Columns:**
- id (BIGINT, Primary Key, Auto Increment)
- first_name (VARCHAR(100), NOT NULL)
- last_name (VARCHAR(100), NOT NULL)
- email (VARCHAR(255), UNIQUE, NOT NULL)
- password (VARCHAR(255), NOT NULL - BCrypt hashed)
- enabled (BOOLEAN, DEFAULT TRUE)
- created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP)
- updated_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP)

**Related Tables:**
- roles: Stores available user roles (USER, ADMIN)
- user_roles: Junction table linking users to roles

## API Endpoints

**POST /api/auth/register**
- Purpose: Create new user account
- Request Body: firstName, lastName, email, password
- Success Response: 201 Created with user data
- Error Response: 409 Conflict if email already exists

**POST /api/auth/login**
- Purpose: Authenticate user and generate JWT token
- Request Body: email, password
- Success Response: 200 OK with user data and JWT token
- Error Response: 401 Unauthorized for invalid credentials
